# Deterministic Reliability Improvement Plan (BC-first)

## Goal
Deliver a first-wave hardening pass that significantly reduces flaky behavior and improves determinism while preserving backward compatibility by default.

## Guardrails (Backward Compatibility)
- Keep public class/function signatures stable; only add optional keyword arguments.
- Keep existing defaults and behavior unless explicitly opting into new controls.
- Keep existing result/context keys and CLI behavior; only add metadata fields.
- Avoid breaking changes to abstract contracts in:
  - `scythe/core/ttp.py`
  - `scythe/behaviors/base.py`
  - `scythe/auth/base.py`
  - `scythe/journeys/base.py`
  - `scythe/orchestrators/base.py`

## Phase 1: Deterministic Foundation + Correctness Quick Wins
### Files
- `scythe/ttps/web/sql_injection.py`
- `scythe/ttps/web/uuid_guessing.py`
- `scythe/core/executor.py`
- `scythe/journeys/base.py`
- `scythe/journeys/actions.py`
- `scythe/ttps/web/request_flooding.py`

### Changes
- Fix mutable default input payload in SQL injection TTP (`full_form_payload` should not be a shared dict).
- Fix UUID status-code check (`if resp.status_code in (404, 401, 403)`).
- Introduce optional deterministic controls (clock/sleep/RNG injection points) in runtime paths that currently call `time.time()`, `time.sleep()`, or global randomness.
- Keep production defaults unchanged (real time + default RNG) unless deterministic controls are provided.

### Why first
- Removes known correctness and shared-state hazards quickly.
- Creates deterministic primitives needed for stable tests in later phases.

## Phase 2: Retry Transparency (No More Hidden Instability)
### Files
- `scythe/orchestrators/batch.py`
- `scythe/journeys/actions.py`
- `scythe/orchestrators/scale.py`

### Changes
- Refactor retry handling so first-failure and retry-failure/success are explicitly recorded.
- Add additive result metadata (`attempt_count`, `retry_reason`, `final_error_type/message`, `retry_wait_s`).
- Keep existing top-level success semantics; improve observability without schema breakage.

### Why
- Retries currently mask intermittent failures and make flaky behavior hard to diagnose.

## Phase 3: State Isolation Across Replications (Primary Flake Driver)
### Files
- `scythe/journeys/base.py`
- `scythe/journeys/actions.py`
- `scythe/orchestrators/scale.py`
- `scythe/orchestrators/batch.py`
- `scythe/orchestrators/distributed.py`

### Changes
- Ensure per-replication isolation of mutable state for Journey/Step/Action execution data.
- Stop mutating action instance fields during execution (notably `ApiRequestAction` request data path); use local per-execution copies.
- Prevent shared Journey context mutation across parallel/distributed runs.
- Ensure parallel result aggregation is deterministic (stable ordering key where needed).

### Why
- Current orchestrators pass shared objects into parallel workers, allowing cross-run contamination.

## Phase 4: Distributed Determinism and Context Safety
### Files
- `scythe/orchestrators/distributed.py`
- `scythe/orchestrators/base.py` (if helper utilities are needed)

### Changes
- Replace shared context writes with execution-scoped context and deterministic merge/reduce.
- Make merge precedence explicit and stable (documented and tested).
- Keep distributed orchestrator public API unchanged.

### Why
- Distributed mode currently mutates shared journey context (`distributed_*` keys), which is race-prone.

## Phase 5: Logging Side-Effect Removal + CI Reliability Gate
### Files
- `scythe/core/executor.py`
- `scythe/journeys/executor.py`
- `.github/workflows/pypi.yaml`
- `.github/workflows/create_release.yaml`
- `.github/workflows/tests.yaml` (new)

### Changes
- Remove library-level `logging.basicConfig(...)` and fixed log-file side effects from executors.
- Use module/named loggers only; let callers/CLI configure handlers.
- Add mandatory CI test workflow and make release/publish jobs depend on tests passing.
- Enforce deterministic CI env (e.g., fixed hash seed/timezone).

### Why
- Global logging setup can interfere with tests and embedding applications.
- Release workflows currently publish without a test gate.

## Test Plan (Implementation Verification)
### Unit/integration test updates
- Add deterministic fixtures/helpers in tests for:
  - Seeded RNG
  - Fake clock
  - No-op/controlled sleep
- Update orchestration tests to verify:
  - No cross-replication state bleed
  - Stable outcomes across repeated parallel/distributed runs
  - Retry metadata correctness and non-masking of failures
- Remove/replace real sleeps in tests where possible.

### Commands
- Local focused checks during development:
  - `python -m unittest tests.test_orchestrators -v`
  - `python -m unittest tests.test_behaviors -v`
  - `python -m unittest tests.test_journeys -v`
- Full suite before merge:
  - `python -m unittest discover -s tests -p "test*.py" -v`
- Determinism confidence run (repeat same subset multiple times):
  - `python -m unittest tests.test_orchestrators -v` (run repeatedly in CI job matrix or loop)

### CI requirements
- New `tests.yaml` workflow runs on PR/push.
- `create_release.yaml` and `pypi.yaml` must require successful test job before release/publish.

## Execution Order and Scope Control
1. Phase 1 (quick wins + deterministic primitives)
2. Phase 2 (retry observability)
3. Phase 3 (state isolation refactor)
4. Phase 4 (distributed determinism)
5. Phase 5 (logging + CI gate)

## Out of Scope for This Wave
- Large public API redesigns.
- Any intentional breaking changes to public contracts.
- Broad onboarding/docs UX overhaul (can be a second wave after reliability stabilization).
